# リファクタリング & コード品質 — Vibe Coding 時代の心得

> リファクタリング＝**外から見た動きは変えず**に、中身を整理して「**読みやすく・直しやすく**」すること。
> ゴールは **人間（と AI）が短時間で把握できる**コード。合言葉は「**動きは変えない**」——テストを通しながら小さく刻む。

---

## 1. 基本の原則（まず「人間が把握できる」コードに）

### 🧪 テストできる形に書く

- 「テストを書く」→ **「テストが“書ける”ように書く／直す」**。
- テストしにくい＝密結合・副作用まみれのサイン。**テストのしやすさ＝設計の良さの物差し**。
- コツ：入力→出力が決まる**純粋関数**に切り出す／DB・API・時刻・乱数などの外部依存は**引数で渡す**（中で直に呼ばない）。

### ♻️ 重複を消す（DRY）

- コピペ＝**関数化**／似た処理＝**共通化して関数化・抽象化**。
- 同じ知識が2か所にあると**直し忘れでバグる**。「1つの知識は1か所」に集約。
- ⚠️ 「たまたま似てるだけ」を無理に1つにしない（**早すぎる抽象化**に注意）。目安は「3回目に出たら共通化を検討」。

### 📏 小さく保つ

- **1関数 20行以下 / 1ファイル 400行以下**（目安）。
- 人間が**ひと目で把握**できる大きさに。長い＝**責務を持ちすぎ**のサイン。
- コツ：長い関数は「意味のまとまり」で小関数に分け、**良い名前で説明**する（コメントより良い名前）。

### 🔌 疎結合にする（依存をなくす）

- 部品どうしの絡みが少ないほど、**1か所の変更が他へ波及しない**＝安全に直せて、テストもしやすい。
- コツ：直接 `new` せず**引数で受け取る**（依存性注入）／具体でなく**“約束（interface）”**に依存する。

### 🔀 条件を整理する

- 深いネスト・複雑な `if` は**読み違い＝バグの温床**。
- コツ：**早期 return（ガード節）**で入れ子を浅く／条件は**名前付き**にする（例：`const isAdult = age >= 18`）。

---

## 2. Vibe Coding 時代に、さらに効くこと

AI が大量のコードを速く書く時代。上の原則は**さらに重要**になり、加えて次が効きます。

### 🤖 AI の出力を鵜呑みにしない（human-in-the-loop）

- **AI は自信満々に間違える**。diff を読む・テストで守る・公式 doc で裏取り。**未読でマージしない**。

### 🧷 型を効かせる（`any` を避ける）

- 型は**人にも AI にもガードレール**。型があると、AI の取り違え・幻覚（それっぽい嘘）が**早期に落ちる**。

### 🧠 AI が読める形にする＝「コンテキストを渡す」

- 小さく・良い名前・型付き＝**AI も把握しやすい**。人間向けの整理は、そのまま AI 向けになる。
- `CLAUDE.md` / `AGENTS.md` や doc で**文脈と決まりを渡す**。コードは人と AI 両方の読み物。

### 🌱 枯れた定番を選ぶ／構成を先に指定

- 情報が多い（＝枯れた）技術ほど **AI は正確**。マイナー技術は幻覚が増える。
- **DRY のために「ライブラリ構成」を先に指示**すると、AI の共通化が的確になる。

### 🪓 小さく刻む（細粒度コミット／PR）

- AI は速く大量に書く → **差分を小さく・こまめにコミット** → レビュー可能＆すぐ戻せる。
- 大改造を一気にやらせない。1コミット＝1つの意図。

### 🧾 prompt＝context も git で管理／「なぜ」を残す

- AI に渡す md・指示も**ソースの一部**。**すべて git 管理**（text がすべて）。
- AI は **What は書けるが Why は残らない** → 非自明な理由はコメントに、経緯は PR 説明・決定記録に**意図を残す**。

### 🛡 セキュリティを疑う（鍵・注入・依存）

- AI は**鍵を直書き**したり、危険な依存を足したりしがち。**秘密・インジェクション・サプライチェーン**を必ず点検。
- 「動く」で満足しない。“**ここが危ない／ここが安全**”を読める目を持つ。

### ✂️ やりすぎ注意（YAGNI）

- AI は**過剰設計・重複**をしがち。要らない抽象・使わない汎用化は削り、**シンプル**に保つ。

## 3. 「気をつける」をやめて、ルールで自動化

原則は根性では続かない。**linter / formatter / CI** に落として、**機械に指摘させる**。レビューが「人の好み論争」でなく「客観ルール」になります。

- **ESLint**：`max-lines-per-function`（関数の長さ）／`max-lines`（ファイル）／`complexity`（複雑さ）／`max-depth`（ネスト）／`@typescript-eslint/no-explicit-any`。
- **重複検出**：`jscpd` などでコピペを可視化。
- **Prettier**：整形は自動（好みの議論をなくす）。
- **CI**：`lint` / `typecheck` / `test` が**通らなければマージをブロック**。
- **ルールは最初はゆるく、少しずつ厳しく**。

---

## 4. 進め方（安全なリファクタリングの順番）

1. **テストで「いまの動き」を固定**する（回帰テスト）。
2. **小さく直す**（1つの意図だけ）。
3. **テストが緑のまま**を確認 → コミット。
4. くり返す。

> 迷ったら：**「これは “動き” を変えていないか？」** と自問する。動きを変えるなら、それはリファクタリングでなく **機能変更**。分けて行う。
