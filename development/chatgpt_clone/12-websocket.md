## 📱 ChatGPTではこう見える

ChatGPTを使っていると、こんな場面に出会います。

- 返事が長すぎたとき、**「⏹ 停止」ボタン**を押すと、生成が **途中でピタッと止まる**
- グループで使うチャットだと、相手が打っている間 **「入力中…」** とふわっと出る

どちらも共通点があります。**こちら（ブラウザ）からサーバーへ、返事の最中でも“合図”を送っている**こと。前章のSSE（流しっぱなしの放送）だけでは、ちょっと工夫が要ります。この章で、その正体をのぞいていきます。

---

## 🤔 一方向（SSE）と双方向（WebSocket）のちがい（🟢 基礎）

第11章でやった **SSE（Server-Sent Events＝サーバー送信イベント）** は、**サーバー → ブラウザの一方向**でした。返事を1文字ずつ「流して送る」のは得意。でも、流れている最中に **ブラウザ → サーバー**へ何かを言うのは、その通り道では基本できません。

そこで出てくるのが **WebSocket（ウェブソケット）**。こちらは **双方向**、つまり **ブラウザ⇄サーバーが、いつでも互いに送り合える** 1本のつなぎっぱなし（常時接続）の線です。

> 🧰 **たとえ話：トランシーバーと放送**
> SSEは **一方通行の放送（ラジオ）**。アナウンサー（サーバー）の声は届くけれど、聞いている側（ブラウザ）は同じ電波で話し返せません。
> WebSocketは **トランシーバー**。ボタンを押せば **どちらからでも** しゃべれます。だから「いま止めて！」「タイピング中だよ」を **会話の最中に**送り合えるのです。

だから、**生成を途中で止める／入力中を知らせ合う**ような“返事の最中の合図”には、双方向のWebSocketが向いています。

### でも——多くのチャットはSSEで足りる（🟢 基礎）

ここが大事な温度感です。「双方向＝えらい」ではありません。

- 1人で使う、**送る → 流れて返ってくる** だけのチャットなら、**SSEで十分**。実装もシンプルです。
- **本当に双方向が要るとき**——リアルタイムに止める、複数人の入力中をリアルタイム共有する、サーバーから突然プッシュする——そういうときに **はじめてWebSocketを検討**します。

> 📌 合言葉：**「まずSSEで作る。双方向が本当に必要になったら、そこで初めてWebSocketを足す。」**
> ちなみに「止める」だけなら、SSE＋別経路（普通のHTTPで“stop”を送る）でも実現できます。WebSocketは“常時つなぎっぱなしで合図し合う”のが本領です。

## 🛠 こう作る（🟢 基礎・紹介）

雰囲気だけ、ごく短い例で。Node.js でよく使う **`ws`** というライブラリで、「**つなぐ → メッセージを受け取る → "stop" が来たら中断**」の骨組みを見てみます。

```ts
import { WebSocketServer } from "ws"; // 双方向通信のための道具

const wss = new WebSocketServer({ port: 8080 });

wss.on("connection", (socket) => {        // ① ブラウザが1本つないできた
  const stop = new AbortController();      //    “中断スイッチ”を用意

  socket.on("message", (raw) => {          // ② ブラウザから合図が届いた
    const msg = JSON.parse(raw.toString());
    if (msg.type === "chat") streamReply(socket, msg.text, stop.signal);
    if (msg.type === "stop") stop.abort();  // ③ "stop"なら生成を打ち切る
  });
});
```

**1行ずつ読むと：**
- `new WebSocketServer({ port: 8080 })`：双方向の **待ち受け口**を1つ開く（ポート＝接続の入り口の番号）。
- `wss.on("connection", (socket) => …)`：ブラウザが**つないでくる**たびに、その人専用の `socket`（話し相手1本）が手に入る。
- `const stop = new AbortController()`：**中断スイッチ**を用意（`AbortController`＝「やめて」を伝えるための標準の仕組み。詳しくは付録F）。
- `socket.on("message", …)`：その線から**合図が届く**たびに呼ばれる。放送（SSE）と違い、ここで**ブラウザ側の声を受け取れる**のがミソ。
- `if (msg.type === "stop") stop.abort()`：合図が `"stop"` なら、スイッチを入れて **生成を打ち切る**。

これだけで「会話の最中に止める」の入り口が見えます。ただし、ここはあくまで**紹介**です。

> 💡 本格的な実装（再接続、複数ユーザーの管理、認証つきの接続）は **付録E**へ。**「止める」を押した後の後始末**（途中まで出た返事をどう保存するか・履歴を壊さないか）は **付録F** にまとめています。この章では「双方向だと“止める／入力中”が素直に書ける」という**感覚**がつかめればOKです。

## ⚠️ ハマりどころ

- **つなぎっぱなしの管理を忘れる**：WebSocketは線を**張りっぱなし**にします。回線が切れたときの **再接続**や、閉じたソケットの **後片付け**を考えないと、いつの間にか繋がらない／無言になります。
- **状態がサーバーに溜まる**：「誰がどの線か」をサーバーが**覚え続ける**ため、利用者が増えた本番（スケール時）で効いてきます。サーバーを複数台に増やすと「相手がどの台にいるか」問題が出ます（付録E）。
- **SSEで足りるのにWebSocketを使う（過剰設計）**：双方向が要らない普通のチャットにまでWebSocketを持ち込むと、**配線が増えて壊れやすく**なります。要るときだけ。
- **認証をWebSocketでもやる必要**：背骨②（🛡守り）はここでも生きています。HTTPで守っていても、**WebSocketの接続そのものにも「誰？」の確認**が要ります（つなぎっぱなしだからこそ、最初の接続時にチェック）。詳しくは付録E。

## 📝 ことばメモ

- **WebSocket（ウェブソケット）**：ブラウザとサーバーをつなぎっぱなしにし、**双方向**でメッセージを送り合える通信のしくみ
- **双方向**：ブラウザ→サーバーも、サーバー→ブラウザも、どちらからでも送れること（SSEは一方向）
- **常時接続**：1本の線を**張りっぱなし**にして、必要なときにすぐ送り合える状態
- **接続管理**：誰がつながっているか、切れたらどうするか（再接続・後片付け）を面倒みること。利用者が増えるほど重要
