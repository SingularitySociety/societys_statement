# フロントエンドの作り方 — 素のJS/React/Vue と SPA/SSR/SSG

> このページは入口（ハブ）です。**まずここを読んで、上から順に各ページへ進んでください。**

Webアプリの「見た目と操作」を作るのがフロントエンド。でも入口でいきなり、**素のJavaScript・React・Vue・SPA・SSR・Next.js…** と用語の壁が立ちはだかります。「結局どれで作ればいいの？」「“React がバックエンドでも動く”ってどういうこと？」——このモヤモヤを、完全未経験の人でも腹落ちするように整理します。

姉妹編 **[コンピューティングの選択肢](../compute_options/README.md)（どこで動かす）** と **[データの置き場所ガイド](../data_stores/README.md)（どこに置く）** に続く、**「画面を どう作り・どこで描くか」**の巻です。

## いちばん大事な考え方：2つの問い 🧭

フロントエンドの選択肢は多いですが、**たった2つの問い**に分けると一気に見通せます。

```
問い①：何で作る？（命令的 → 宣言的）
   素のTS/JS … 自分で画面を手作業で書き換える（命令的）
   React / Vue … 「状態がこうなら画面はこう」と書くと、変化は自動で反映（宣言的・リアクティブ）

問い②：どこで描画する？（ブラウザ ↔ サーバー）
   SPA(CSR) … ブラウザのJSが画面を組み立てる（アプリ的・サクサク／初回は重い・SEO弱め）
   SSR/SSG  … サーバー側でHTMLを用意（初回が速い・SEO強い）
   ハイブリッド … ページごとに混ぜる（Next.js など。“一部だけサーバー”もできる）
```

> 💡 この2軸さえ持っておけば、新しい名前が出てきても「あ、これは①が宣言的で、②がサーバー寄りね」と**地図の上に置ける**ようになります。

## 🍳 たとえで早見

| | たとえ | ひとことで |
|---|---|---|
| **素のTS/JS** | 手で人形を動かす | 自分でDOMを操作（命令的）。小さい用途なら十分 |
| **React / Vue** | 「今は喜び」と書けば笑顔になる | 状態を書けばUIが追従（宣言的）。複雑な画面に強い |
| **SPA（CSR）** | 材料を送って客が自分で調理 | 最初は待つが、以後の操作はサクサク |
| **SSR** | 厨房で作って皿で出す | すぐ食べられる（初回が速い・SEO○）→あとでJSで“動く”化 |
| **SSG** | 作り置きを出す | 最速・最安・安全。内容が頻繁に変わらないもの向け |

## 🔑 最初に覚える勘どころ

- 🔒 **SPAでもAPIキー・秘密はフロントに出さない。** SSR や React Server Components は「**一部をサーバーで動かせる**」ので、秘密をサーバー側に置けるのが利点（姉妹教材の「鍵を守る」と地続き）。
- 🔎 **SEO・初回の速さが要るなら SSR/SSG、アプリ的な操作中心なら SPA。**
- 🌱 **まず素のHTML/JSで動かし、必要になってからフレームワーク。** 最初から全部盛りにしない。
- 🤝 姉妹教材（Twitter/ChatGPTクローン）が**素のTS/JS**で書いたのは「仕組みに集中するため」。実務では React/Vue が主流——その理由もこのガイドで分かります。
