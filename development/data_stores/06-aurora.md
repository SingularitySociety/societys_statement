# Aurora — クラウドネイティブな高性能・高可用RDB

> 📖 このページのゴール：前ページの **RDS** の「もう一段上」にあたる **Aurora（オーロラ）** が何者で、RDS と何がどう違うのかを、たとえと表で腹落ちさせる。これで「SQL系をどう動かすか（運用のはしご）」の右側が見えてきます。
> [← 目次・はじめにへもどる](README.md)

---

## 🍳 ひとことのたとえ

前ページの **RDS** は「**料理（DB）は自分のレシピ、キッチンの掃除や火の管理はお店（AWS）にお任せ**」でした。Aurora は、その**ひとつ上のグレード**です。

- 出てくる料理は**同じ**（**MySQL / PostgreSQL 互換**＝食べる人＝アプリからは「いつものあの味」）。
- でも、**キッチンそのものを“クラウド専用”に作り直した高性能版**。普通の家庭用コンロではなく、業務用の超高速・大容量・自動消火つきの厨房です。

つまり Aurora とは——

> 🟢 **「同じ料理（MySQL/PostgreSQL 互換）を、クラウド専用に作り直した高性能キッチンで出す」RDB。**
> アプリから見れば「いつもの SQL」。裏側のキッチン（ストレージのしくみ）だけが、別物の高性能・高可用版に置き換わっている、と覚えてください。

---

## 🧩 何者？（しくみ）

**Amazon Aurora（アマゾン・オーロラ）** は、AWS が独自開発した **クラウドネイティブ**な RDB です。

- **クラウドネイティブ**（＝クラウド前提で一から設計）：既存の DB をそのままクラウドに載せたのではなく、**「クラウドで動かすこと」を前提に中身を作り直した**ということ。
- **MySQL / PostgreSQL 互換**：アプリのコードや SQL は**ほぼそのまま**。「いつもの MySQL/PostgreSQL のつもりで使える」ので、乗り換えの心理的ハードルが低い。

ここからが、ふつうの DB と違う Aurora の“肝”です。

### ① ストレージとコンピュート（計算）を「分離」した

ふつうの DB は、**計算する部分**と**データを置く部分**が同じサーバーに同居しています。Aurora は、この2つを**切り離し**ました。

```
ふつうの DB:   [計算 ＋ データ] が同じ1台に同居
                  ↑ ここが詰まると全部が遅くなる／壊れると全部困る

Aurora:        [計算] ────→ [みんなで共有する高速・自動拡張ストレージ]
               （頭脳）          （巨大な共有倉庫。複数の場所に自動コピー）
```

- **頭脳（計算）と倉庫（ストレージ）を別々に増やせる**ので、ムダがなく、速くて丈夫。
- 倉庫は AWS が用意した**専用の高速ストレージ**。ここが Aurora の性能・可用性の源泉です。

### ② 高可用：複数の場所（AZ）に自動でコピー

- **AZ**（エーゼット＝アベイラビリティーゾーン）＝**離れた場所にあるデータセンター**のこと。
- Aurora はデータを、**複数の AZ に自動で何個もコピー**して持ちます。
- だから、ある建物（AZ）が丸ごと落ちても、**別の場所のコピーで動き続けられる**＝**高可用（こうかよう＝めったに止まらない）**。

### ③ 高速＆ストレージ自動拡張

- 読み書きが速くなるように、上記の専用ストレージが工夫されている（**高速**）。
- データが増えても、**倉庫が自動でひとりでに広がる**（**ストレージ自動拡張**）。「容量が足りない！」と慌てて手当てする作業が、基本いりません。

### ④ Aurora Serverless v2：DBが負荷で“伸び縮み”する

- **Aurora Serverless v2（サーバーレス・ブイツー）** ＝ **アクセスの多い・少ないに合わせて、DBの能力が自動でスケール（増減）する**しくみ。
- 忙しい時間帯は自動でパワーアップ、**使わない時は縮んで**コストを抑える。
- 「夜はガラガラ・昼はピーク」のような**波のあるアプリ**と相性がよい。

> 🔧 ここでの「サーバーレス」は、姉妹編 [コンピューティングの選択肢](../compute_options/README.md) で出てくる「サーバーを意識しない」の感覚と同じ。**サーバーが無いのではなく、容量を自分で決めなくていい**＝伸び縮みをお任せできる、という意味です。

## 🛠 代表サービス（実名）

| 名前 | 何 | ひとこと |
|---|---|---|
| **Amazon Aurora（MySQL 互換）** | AWS のクラウドネイティブ RDB | いつもの MySQL のつもりで、高性能・高可用に |
| **Amazon Aurora（PostgreSQL 互換）** | 同上の PostgreSQL 版 | Postgres 派はこちら。機能も Postgres 寄り |
| **Aurora Serverless v2** | 負荷で自動伸縮する Aurora | 波のある負荷・使わない時は縮めたい人に |

> 🔧 立ち位置：Aurora は **AWS 専用**のサービスです。「他社にも同じ Aurora がある」わけではありません（後述のロックインに直結）。他社の“高性能クラウドDB”としては、Google の **AlloyDB**（Postgres 互換）などが似た発想の製品としてあります。

---

## 🆚 RDS と Aurora、何が違う？（ここが本題）

どちらも「AWS にお任せのマネージド RDB」ですが、**中身のキッチンの作り**が違います。RDS が「ふつうの DB を AWS が運用代行」なのに対し、Aurora は「**AWS がクラウド用に作り直した別物の高性能版**」。

| 評価軸 | 🍳 RDS（前ページ） | 🍱 Aurora（このページ） |
|---|---|---|
| **中身** | ふつうの MySQL/PostgreSQL 等を、そのまま運用代行 | クラウド用に**作り直した**MySQL/PostgreSQL **互換**エンジン |
| **性能** | そのDB本来の性能（十分速い） | **より高速**（専用ストレージで読み書きが効率的） |
| **可用性（止まりにくさ）** | フェイルオーバー対応で十分高い | **さらに高い**（複数AZへ自動コピー・復旧が速い） |
| **スケール（規模拡大）** | 主に**垂直**（マシンを大きく）＋読み取りレプリカ | **読み取りの拡張に強い**・ストレージ**自動拡張**・Serverless で自動伸縮 |
| **価格** | 比較的おさえめ | **やや高め**（高性能・高可用のぶん） |
| **ロックイン** | 中（中身は普通のDBなので比較的移りやすい） | **強め**（AWS 専用。引っ越しはやや大変） |
| **共通点** | マネージド（パッチ・バックアップ・復旧は AWS） | 同じくマネージド。**SQL もアプリもほぼそのまま** |

> 🟢 ざっくり：**「とりあえず素直に・安く」なら RDS、「高負荷でも速く・絶対に止めたくない」なら Aurora。** どちらも SQL なので、アプリ側の書き方は大きく変わりません。

## 👍 向いている / 👎 向かない

| | こんな時に強い 👍 | こんな時はやめておく 👎 |
|---|---|---|
| 負荷 | **高負荷**・アクセスが多いサービス | **小規模**で、まだアクセスがごく少ない |
| 可用性 | **絶対に止めたくない**（決済・基幹など） | 多少止まっても困らない・趣味アプリ |
| 規模 | **成長中〜大規模**でこれから伸びる | コストを1円でも切り詰めたい初期段階 |
| 読み書き | **読み取りが多い**（読み取り拡張が得意） | データもアクセスもごく軽い |
| ロックイン | AWS を本格的に使う前提でOK | **AWSに縛られたくない**・将来よそへ移すかも |

> 🟢 言い換えると、Aurora は **「RDS では少し物足りなくなってきた、成長中〜大規模アプリ」**のための一段上の選択肢。小さいうちは **RDS（や Supabase/Neon）で十分**で、Aurora は**過剰**になりがちです。

---

## 💰 コスト感

- **RDS より高め**。高性能・高可用というぶん、相応のお値段になります。
- 小規模・低トラフィックだと、**性能を持て余して“割高”**になりがち。「速さと止まりにくさ」に**お金を払う価値がある規模**になってから。
- **Aurora Serverless v2** は**従量（使った分だけ）**寄り。波のある負荷なら、**使わない時に縮んでコストを抑えられる**のが利点。ただし常時フル稼働だと、必ずしも安くはなりません。

> 🟢 目安：**「止まると損が大きい」「速さがお金を生む」規模になったら、コスト増に見合う。** そうでないうちは RDS で十分です。

---

## 🔧 運用/メンテ

**RDS と同じく“ほぼお任せ（マネージド）で楽”**。パッチ当て・バックアップ・障害時の切り替え（フェイルオーバー）は、基本 AWS が面倒を見ます。むしろ Aurora は、

- **復旧が速い・止まりにくい**ぶん、夜中に叩き起こされる回数が減りやすい。
- ストレージは**自動拡張**なので、「容量が足りない！」の手当てがいらない。

——と、**運用のラクさはむしろ上**。あなたの仕事は「**どのデータを・誰に許すか（設計とアクセス権）**」に集中できます（これはどの DB でも変わらない本体の仕事）。

## 🏢 誰が使う？

| ペルソナ | 相性 | ひとこと |
|---|---|---|
| **個人** | △ | ふつうは過剰。まず **RDS / Supabase / Neon** で十分 |
| **スタートアップ** | ○（伸びてきたら） | 最初は軽量に。**スケールが見えてきたら**乗り換え候補 |
| **中堅ベンチャー** | ◎ | **高負荷・高可用が必要**になった本番の主力 DB に最適 |
| **大企業** | ◎ | 基幹・大規模で、**止まらない・速い**を満たす定番のひとつ |

> 🟢 タイミングが肝：Aurora は「**最初に選ぶ DB**」ではなく、「**成長してスケールが見えてきたら検討する DB**」。中堅〜大企業、あるいは急成長中のサービスの“次の一手”です。

## 🟢 ひとことで言うと

Aurora は、**RDS の上位グレード**。出てくる料理（MySQL/PostgreSQL 互換）は同じなのに、**キッチン（ストレージのしくみ）をクラウド専用に作り直した高性能・高可用版**です。**ストレージとコンピュートを分離**し、複数AZへ自動コピー・自動拡張・**Serverless v2 で自動伸縮**。**高負荷・高可用が必要な成長中〜大規模アプリ**にうってつけ。ただし**小規模には過剰・割高**で、**AWS ロックインが強め**なのが注意点。**迷う規模ならまず RDS、止まると損が大きくなってきたら Aurora**、と覚えておけば大丈夫です。
