## 🟢 まず始め方：いきなり全部やらない

CI/CD と聞くと「テスト・ビルド・セキュリティ・デプロイ…全部いっぺんに組むの？」と身構えがちですが、**逆です**。鉄則はこれだけ。

> 🟢 **まず `.github/workflows/ci.yml` を1つ作って、「push でテストを走らせる」だけ。** 小さく始めて、効いてきたら少しずつ足す。

| ステップ | やること | 足すきっかけ |
|---|---|---|
| 🟢 **第0歩** | `.github/workflows/ci.yml` を作り、**push でテストだけ**自動化 | いますぐ |
| 🟢 第1歩 | PR でもテスト＋**lint/型チェック**を追加、**必須チェック**にしてマージをブロック | チームで開発し始めたら |
| 🔧 第2歩 | **CodeQL**（セキュリティ）／**Dependabot**（依存更新）を足す | 公開・本番が見えてきたら |
| 🔧 第3歩 | **デプロイ／リリース**（CD）を自動化 | 「手で届ける」のが面倒になったら |
| 🔧 第4歩 | **マトリクス**（複数OS/版）・**キャッシュ**で速く・広く | 規模が大きくなったら |

ポイントは、**最初の1ファイルが動いた時点で、もう CI/CD は始まっている**こと。完璧を目指して固まるより、**小さく動かして育てる**のが正解です。

## 🧭 選び方フローチャート：何を自動化したい？

「結局、自分は何を自動化すればいいの？」を、**やりたいこと起点**でたどれる地図です。

```
Q. 何を自動化したい？
│
├─ 🟢「壊れてないか確認したい」（まずここから）
│     ├─ テストを毎回回す ……… on: push / pull_request +「npm test」
│     ├─ コードの体裁・型を見る … lint / tsc を step に追加
│     └─ 脆弱性を見つけたい …… 🔧 CodeQL（コードスキャン）
│
├─ 🔧「できたものを自動で届けたい」（CD）
│     ├─ Webサイト/アプリを公開 … Vercel / Netlify / GitHub Pages へデプロイ
│     ├─ バージョンを配布したい … タグ push → GitHub Release（成果物を添付）
│     └─ サーバーへ送りたい …… SSH/rsync・Docker イメージを push
│
├─ 🔧「決まった時間に動かしたい」
│     └─ 毎朝バッチ・リンク切れチェック … on: schedule（cron＝定期実行）
│
└─ 🔧「PRレビューを楽にしたい」
      ├─ 機械に一次レビューさせる … レビューBot（CodeRabbit / Sourcery / Copilot）
      └─ 依存ライブラリを自動更新 … Dependabot（更新PRを自動で作る）
```

> 🟢 迷ったら**いちばん上のルート**（push でテスト）から。CD やレビューBot は、土台ができてから足せば十分です。

## ⚖️ 比較早見表 その1：手作業 vs CI/CD

「そもそも自動化って、手でやるのと何が違うの？」を4つの軸でまとめます。

| 軸 | 😱 手作業 | 🏭 CI/CD（自動） |
|---|---|---|
| **ミス** | 忘れる・サボる・手順を1つ飛ばす | **毎回・同じ**。うっかり飛ばしが起きない |
| **速さ** | 1つずつ順番に、人が張り付く | **並行**で回り、待ち時間に他の作業ができる |
| **安心** | 壊れていても気づかずマージ／リリース | 壊れていれば**即ブザー**、マージをブロックできる |
| **再現性** | 「自分の環境では動く」が起きる | いつも**まっさらな同じ環境**。誰がやっても同じ |

> 🟢 ひとことで：**「人の気合い」ではなく「しくみ」で品質を守る**——気合いは続かないが、しくみは続く。

## 🎬 トリガー早見表：いつ動かす？（`on:`）

ワークフローは「**いつ動かすか**＝トリガー」で性格が決まります。代表的な5つを「何に使う？」とセットで。

| トリガー（`on:`） | 読み | いつ動く？ | 何に使う？ |
|---|---|---|---|
| `push` | プッシュ | コードを push したとき | 🟢 **まずこれ**。テスト/ビルドの自動チェック |
| `pull_request` | プルリクエスト | PR が作られた／更新されたとき | 🟢 マージ前の最終チェック（必須チェック化） |
| `schedule` | スケジュール | **cron** で決めた時刻に定期的に | 🔧 毎朝バッチ・リンク切れ・依存の棚卸し |
| `workflow_dispatch` | — | GitHub の画面の**手動ボタン**を押したとき | 🔧 任意のタイミングで手動実行・再実行 |
| `release` / タグ | リリース／タグ | リリース作成や**タグ push** のとき | 🔧 リリースの自動化（成果物を添付して配布） |

## 👥 ペルソナ別：あなたのタイプ別おすすめ

「どこまでやれば十分？」は規模で変わります。**背伸びしすぎない**目安をどうぞ。

| タイプ | おすすめ構成 | ねらい |
|---|---|---|
| 🟢 **個人開発** | push で**テスト** ＋ Pages/Vercel への**自動デプロイ** | 「テスト忘れ」と「手デプロイ事故」をなくす |
| 🔧 **スタートアップ** | テスト ＋ デプロイ ＋ **レビューBot**（CodeRabbit 等） | 少人数でも**一次レビューを機械に**任せて速度を出す |
| 🔧 **中堅** | **マトリクス**（複数OS/版）＋ **必須チェック** ＋ **CodeQL** ＋ **Dependabot** | 品質を**面で**担保し、依存・脆弱性も自動で追う |
| 🔧 **大企業** | 上に加えて **環境分離**（dev/stg/prod）・**承認フロー**・**監査ログ** | デプロイを**統制**し、誰が何を出したかを残す |

> 🟢 上から下へは**足し算**です。個人で始めて、人や本番が増えたら一段ずつ重ねていけばOK。

## 🌐 GitHub Actions 以外のCI/CDは？

CI/CD のしくみは GitHub Actions だけのものではありません。**GitLab CI/CD**（GitLab 組み込み）・**CircleCI**・**Jenkins**（自前で立てる老舗）など、ツールはいろいろあります。ただし**考え方はどれも同じ**——「**きっかけ（push など）→ 決めた手順を自動で流す**」「設定は**コード（YAML 等）で書く**」という背骨は共通です。だから、ここで覚えた **`on → jobs → steps`** の感覚は他ツールでもそのまま生きます。そのうえで、**コードを GitHub に置いているなら、追加の契約も別サービスもいらず最短で始められる** GitHub Actions が、いちばん手軽な入口になります。

---

## 🟢 ひとことで言うと

**まずは `ci.yml` を1つ作って「push でテスト」だけ**——CI/CD はそこから始まります。あとは🧭フローチャートで「何を自動化したいか」を選び、**手作業 vs CI/CD**（ミス・速さ・安心・再現性）と**トリガー早見表**を地図に、自分のタイプに合わせて**一段ずつ足す**だけ。考え方は他のCI/CDツールでも同じで、GitHub に置いているなら Actions が最短の入口です。
