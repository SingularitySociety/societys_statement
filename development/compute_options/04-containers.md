# コンテナ — アプリを箱に詰めて、どこでも同じに動かす

> 📖 このページのゴール：「コンテナ」を**お弁当箱のたとえ**で腹落ちさせます。「私のパソコンでは動くのに、本番だと動かない」——あの悲しい事故がなぜ起きて、コンテナがどう消してくれるのか。そして **Docker（箱の規格）** と **Kubernetes（箱を束ねる仕組み）** の役割の違い、楽をするための **Fargate / Cloud Run** までを、名前だけでなく「何をするものか」で押さえます。
> [← 目次・はじめにへもどる](README.md)

---

## 🍱 ひとことのたとえ

コンテナは **お弁当箱（または冷凍ピザ）** です。

- **中身を箱に詰める** … アプリ本体だけでなく、「動くのに必要な道具一式」（言語の実行環境・ライブラリ・設定）を**まとめて1つの箱**に入れる。
- **どの台所でも同じ味** … その箱は、あなたのパソコンでも、同僚のパソコンでも、本番のサーバーでも、**まったく同じに**開いて温められる。台所（環境）が違っても、味（動き方）が変わらない。
- **持ち運べる** … 箱ごと別のサーバーへ移しても、また同じに動く。引っ越しがラク。

冷凍ピザで言えば——**どのオーブンに入れても同じピザが焼き上がる**。オーブンのメーカーや古さ（＝サーバーの細かい違い）を気にしなくてよくなる、ということです。

> 🟢 **これが刺さるポイント：「私の環境では動くのに、本番で動かない」問題が消える。**
> 開発者が一度は泣く“あるある”がこれです。原因はたいてい「自分のパソコンと本番で、ライブラリのバージョンや設定が微妙に違う」こと。コンテナは**必要なものを全部箱に同梱する**ので、環境の違いそのものが無くなり、この事故が起きにくくなります。

---

## 🧩 どこまで自分で、どこからお任せ？

前ページの **IaaS（サーバーをまるごと借りて、OS から上は全部自分）** にくらべると、コンテナ（マネージドで使う場合）は **OS の管理あたりまでをクラウドに寄せられる** のが大きな違いです。ただし——

> 🔑 **箱の中身（アプリ＋道具）は、あくまで自分の責任。**

| レイヤー（誰が面倒を見る？） | IaaS | **コンテナ（マネージド）** |
|---|---|---|
| 物理ハード〜仮想化 | クラウド | クラウド |
| OS（土台のシステム） | **自分** | だいたいクラウド |
| ミドルウェア・ランタイム（言語実行環境） | 自分 | **自分（箱の中に詰める）** |
| アプリのコード | 自分 | **自分** |
| **データ・設定・アクセス権（鍵）** | 自分 | **自分** |

ポイントは2つ。
1. **OS の世話が軽くなる** … マネージドサービスを使えば、土台の OS のお守りはクラウド任せにしやすい。
2. **でも“箱の中身”は自分** … 箱に何を入れるか（どのバージョンを使うか、よけいな物を入れないか）は、全部あなた次第。**箱の中で起きることはあなたの責任**です。

---

## 🛠 代表的なサービス（実名）— 「役割」で区別するのがコツ

ここがいちばん混乱しやすいところ。**名前を覚えるより「何をするもの」かで区別**しましょう。

| 名前（読み） | これは何？ | ひとことで |
|---|---|---|
| **Docker（ドッカー）** | 箱の**規格**そのもの | 「お弁当箱の規格」。1個の箱を作って動かす道具 |
| **Kubernetes（クバネティス／k8s）** | 箱を**大量に束ねて回す**仕組み | 「箱の交通整理係」。強力だが**学習が重い** |
| **AWS ECS / Fargate（ファーゲート）** | AWS が**運用を肩代わり**してくれる箱の置き場 | サーバー管理ナシで箱を動かす。**従量** |
| **Google Cloud Run（クラウドラン）** | 箱を置くだけで動く、**サーバーレス的**な実行台 | **ゼロスケール＋従量**。アクセスが無い間は料金もほぼゼロ |
| **Azure Container Apps** | Microsoft 版の手軽な箱の実行台 | Azure 寄りなら有力 |

### 🍱 Docker＝「1個の箱」

まず **Docker** が、箱の規格です。アプリと道具を詰めた**「イメージ」**（箱の設計図）を作り、それを動かすと**「コンテナ」**（実際に動いている箱）になります。多くの場合、**スタートはここ**。1個〜数個の箱なら Docker だけで十分回せます。

### 🚚 Kubernetes（k8s）＝「箱を何百個も束ねる司令塔」🔧

箱が**何十・何百個**に増えると、「どのサーバーにどの箱を置くか」「壊れた箱を自動で立て直す」「急にアクセスが増えたら箱を増やす」といった**交通整理**が必要になります。これを自動でやるのが **Kubernetes**（＝箱の**オーケストレーション**＝大量の箱を束ねて指揮すること）。

- 👍 とても強力（大規模・自動復旧・自動増減）。
- 👎 **学習コストが重い**。**自前で運用するのは大変**で、専任の人手が要りがち。

> 🔧 **楽をするコツ：自前 k8s は避けて「マネージド」にする。**
> Kubernetes を自分で構築・運用するのは骨が折れます。**Fargate** や **Cloud Run** のような**マネージド**（運用をクラウドが肩代わり）を使えば、箱を動かす恩恵だけ受けて、面倒な運用は任せられます。

### 🚰 Cloud Run / Fargate＝「サーバーレス的に箱を動かす」

**Google Cloud Run** と **AWS Fargate** は、「コンテナ（箱）」なのに**サーバーレス的**に使えるのが魅力です。とくに Cloud Run は——

- **従量課金**（使った分だけ）＋ **ゼロスケール**（アクセスが無い間は箱の数が0になり、料金もほぼかからない）。
- サーバーの台数やOSの世話を**考えなくてよい**。

> 💡 「コンテナ」と「サーバーレス」は別物ではなく**地続き**です。Cloud Run は「箱は使うけど、動かし方はサーバーレス寄り」。この“どのへんに立つか”の感覚が、はしご全体を理解するカギになります（詳しくは [サーバーレス・BaaSのページ](06-serverless-baas.md)）。

## 👍 向いている / 👎 向かない

| 👍 向いている | 👎 向かない（過剰になりがち） |
|---|---|
| **移植性**がほしい（どこでも同じに動かしたい） | **小規模・趣味の1個アプリ**（箱や運用が大げさ） |
| **開発と本番を揃えたい**（“動かない”事故を消す） | チームが小さく、**運用に人手を割けない**のに k8s を自前で組む |
| **マイクロサービス**（小さな部品を多数組み合わせる作り） | まずは速く出したいだけ → [PaaS](05-paas.md) のほうが近道なことも |
| **複数の言語・技術**が混在するチーム | 学習の余裕が無い段階で、いきなり k8s に挑む |

---

## 💰 コスト感

| 使い方 | コストの傾向 |
|---|---|
| **Fargate / Cloud Run（マネージド・従量）** | 🟢 手軽。使った分だけで、Cloud Run はゼロスケールで**待機中ほぼ0円**。小さく始めやすい |
| **Kubernetes を自前運用** | 🔧 サーバー代より**人件費（運用・学習）が重い**。クラスタを常時動かすと固定費も乗る |

> 💡 **「安い＝総額が安い」ではない**のが、はしご共通の注意点。Kubernetes は強力ですが、**運用する人の時間**という“隠れコスト”が一番大きくなりがちです。少人数なら、まずマネージドで軽くするのが賢い選択です（→ [コストのページ](07-cost.md)）。

## 🔒 セキュリティ

任せる範囲が広がってOSの世話は軽くなりますが、**箱の中はあなたの責任**です。

- **イメージ（箱の設計図）の脆弱性**に注意 … 古いライブラリや既知の穴を箱に詰めたままにしない。**こまめに更新**する。
- **最小権限** … 箱やサービスに、必要以上の権限・鍵を持たせない。
- **余計なものを箱に入れない** … 使わないツールやデバッグ用の道具を同梱しない。中身が増えるほど穴も増える。
- **データ・設定・アクセス権は最後まで自分** … これは全ページ共通の鉄則。公開設定・権限・鍵の置き場所のミスが、規模を問わず最大の事故原因です（→ [セキュリティのページ](08-security.md)）。

---

## 🔧 長期メンテ

- **イメージの更新が定期作業** … ベースにした OS やライブラリにも提供終了（EOL）や脆弱性が出ます。**箱を作り直して入れ替える**運用が要ります。
- **Kubernetes を自前で持つと運用が重い** … バージョン追従・監視・障害対応が継続的に発生。
- **マネージドで軽くできる** … Fargate / Cloud Run に寄せれば、土台のお守りはクラウド任せにでき、メンテの負担をぐっと下げられます（→ [長期メンテのページ](09-maintenance.md)）。

## 🏢 誰が使う？（4ペルソナ一言）

| 立場 | コンテナとの付き合い方 |
|---|---|
| **大企業** | **主力**。大規模チームでマネージド Kubernetes を使い、多数のサービスを束ねて運用 |
| **中堅ベンチャー** | **主力**。マネージドコンテナでスケールと運用効率を両立。少人数SREでも回せる形に |
| **スタートアップ** | 速度優先なら **Cloud Run が手軽**。箱の恩恵をサーバーレス的に受け、運用は持たない |
| **個人** | **Cloud Run の無料枠**で十分遊べる。月0円〜で「箱で動かす」を体験できる |

## 📝 ことばメモ

- **コンテナ** … アプリと必要な道具をひとまとめにした「箱」。どの環境でも同じに動く。
- **イメージ** … 箱の**設計図**（テンプレート）。これを動かすとコンテナになる。
- **Docker（ドッカー）** … 箱の**規格・道具**。まずはここから。
- **Kubernetes（k8s／クバネティス）** … 箱を**大量に束ねて指揮**する仕組み。強力だが学習が重い。
- **オーケストレーション** … たくさんの箱の配置・増減・復旧を**自動で取り仕切る**こと。
- **マネージド** … 運用（更新・障害対応）をクラウドが**肩代わり**してくれること。
- **ゼロスケール** … アクセスが無い間は箱の数が**0**になり、料金もほぼかからない仕組み。

---

## 🟢 ひとことで言うと

コンテナは「アプリ＋必要な道具」を**お弁当箱に詰めて、どこでも同じに動かす**仕組み。最大のごほうびは **「私の環境では動くのに本番で動かない」問題が消える** こと。**Docker＝箱の規格**、**Kubernetes＝箱を束ねる司令塔（強力だが重い）**、そして **Fargate / Cloud Run＝運用を任せて楽にする** ——役割で区別すれば迷いません。小規模には大げさなこともあるので、**まずはマネージドで軽く**が安全策です。
