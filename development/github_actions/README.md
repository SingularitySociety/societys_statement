# GitHub ActionsではじめるCI/CD

> このページは入口（ハブ）です。**まずここを読んで、上から順に各ページへ進んでください。**

コードを書いたあと、毎回こんなこと、していませんか？——**テストを手で走らせて、ビルドして、問題なければサーバーにアップして…**。地味だけど大事で、しかも**忘れたりミスったりしやすい**作業です。

これを **コンピュータに毎回・自動でやらせる** のが **CI/CD**。そして、その自動化を **GitHub の中だけで** 始められるのが **GitHub Actions（ギットハブ・アクションズ）** です。このガイドは、**「GitHub Actions で何ができて、何が嬉しいのか」**を、完全未経験の人でも腹落ちするように説明します。

姉妹編 **[コンピューティングの選択肢](../compute_options/README.md)（動かす）・[データの置き場所](../data_stores/README.md)（置く）・[フロントエンドの作り方](../frontend/README.md)（作る）** に続く、**「安全に・自動で“届ける”」**の巻です。

## 🏭 いちばん大事な考え方：ベルトコンベア

CI/CD のイメージは **工場のベルトコンベア** です。

```
あなた：コードを commit / push
   │
   ▼   （ここから先は、ぜんぶ自動）
[ テスト ] → [ ビルド ] → [ 検査(lint/型/セキュリティ) ] → [ 出荷(デプロイ/リリース) ]
   │
   ▼
 壊れていれば即お知らせ／問題なければ本番へ
```

- **CI（継続的インテグレーション）**＝コードを上げるたびに、**自動でテスト・ビルドして「壊れてないか」を確認**。
- **CD（継続的デリバリー／デプロイ）**＝確認が通ったら、**自動で届ける（デプロイ・リリース）**。

人は「**中身を作ること**」に集中できて、退屈で間違えやすい作業はコンベアが引き受ける——これが自動化のうれしさです。

## 😊 何が嬉しい？（4つ）

| | うれしいこと | なぜ |
|---|---|---|
| ① | **ミスが減る** | 人は忘れる・サボる。機械は毎回・同じようにやる |
| ② | **速い・ラク** | 待たずに並行、手作業から解放される |
| ③ | **安心してマージ／リリースできる** | 壊れていればすぐ気づく（壊れたコードを入れない） |
| ④ | **チームの品質が揃う** | 「自分の環境では動く」問題を防ぐ |

## 🤖 GitHub Actions は「○○したら△△する」係

**GitHub Actions** は GitHub に組み込まれた自動化のしくみ。**追加費用なし**で始められます（無料枠あり）。やることは、リポジトリの `.github/workflows/` に **YAML の手順書**を置くだけ。

> 「**push されたら** テストする」「**PRが来たら** レビューBotを走らせる」「**毎朝9時に** バッチを動かす」「**タグを付けたら** リリースする」——こういう「**きっかけ → 自動の処理**」を、いくつでも作れます。

## この文書の到達点

最後まで読むと、こんなことができるようになります。

- 「CI」と「CD」の違いと、**自動化のうれしさ**を人に説明できる
- GitHub Actions の **`on → jobs → steps`** の形を読める／小さなワークフローを書ける
- **何が自動化できるか**（テスト・CodeQL・レビューBot・デプロイ・リリース・定期実行・通知）を知っている
- **鍵を安全に渡す**（シークレット・最小権限）勘どころが分かる
